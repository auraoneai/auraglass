## Executive Summary

The redesigned AuraGlass homepage looks substantially better than the older catalog surfaces because it was authored as a single, cohesive marketing composition. It uses hand-rolled CSS and static HTML-like controls to create the impression of AuraGlass components, but many visible elements are not actually rendering package components.

That is acceptable as a prototype, but it is not acceptable as product proof.

AuraGlass 3.0.7 should productize the homepage's strongest visual ideas into a real, reusable **AuraGlass Marketing Kit**:

- Package-owned aurora background system.
- Package-owned hero orb.
- Package-owned display/gradient text.
- Package-owned logo mark.
- Package-owned showcase card.
- Package-owned marketing button variant.
- Package-owned marketing/demo compositions for feature tiles, metrics, install commands, tabs, input rows, and toast rows where appropriate.

The website homepage should then be refactored so the beautiful design remains, but the visible UI is powered by actual `aura-glass` exports rather than custom `.glass`, `.gbtn`, `.ginput`, and fake decorative labels.

3.0.7 should be scoped as a follow-up after 3.0.6. 3.0.6 remains the core stabilization release. 3.0.7 is the first "proof polish" release: the website becomes a canary for whether AuraGlass can build its own premium marketing surface.

## Problem Statement

The current redesigned homepage has a trust gap:

- It looks premium.
- It labels visuals as `<GlassButton />`, `<GlassInput />`, `<GlassTabs />`, `<GlassToast />`, etc.
- But the visible UI is often custom CSS and plain HTML elements, not real package components.

This means the homepage is visually persuasive but technically misleading. A user may reasonably assume AuraGlass can create that exact interface out of the box, while the package currently lacks several homepage-specific primitives.

The fix is not to delete the redesign. The fix is to upstream the design language into AuraGlass so the homepage can honestly use the package.

## Required Package APIs

### 1. `AuroraBackground`

#### Purpose

Provide the redesigned homepage's full-viewport aurora field as a reusable package component.

#### Proposed Location

- Source: `src/components/marketing/AuroraBackground.tsx`
- Styles: `src/components/marketing/AuroraBackground.css` or token-driven inline style helpers consistent with package standards.
- Export: `src/index.ts`
- Docs: `docs/components/marketing/aurora-background.md`
- Tests: `src/components/marketing/AuroraBackground.test.tsx`
- Story: `src/components/marketing/AuroraBackground.stories.tsx`

#### Proposed API

```ts
export interface AuroraBackgroundProps extends React.HTMLAttributes<HTMLDivElement> {
  palette?: 'aurora' | 'prism' | 'ocean' | 'ember' | 'mono';
  intensity?: 'subtle' | 'medium' | 'strong';
  motion?: 'none' | 'subtle' | 'full';
  particles?: boolean | number;
  grain?: boolean;
  vignette?: boolean;
  fixed?: boolean;
  reducedMotion?: boolean;
  seed?: string | number;
}
```

#### Requirements

- SSR deterministic: particle positions must not change between server and client.
- Respects `prefers-reduced-motion`.
- Does not require browser APIs during render.
- Uses package tokens where possible.
- Accepts custom className/style.
- Can run as a fixed background or contained section background.
- Does not block pointer events.
- Keeps GPU cost reasonable.

#### Acceptance Criteria

- The website homepage can replace custom `AuroraBackground.tsx` and `.auroraStage` CSS with package `<AuroraBackground />`.
- Snapshot test verifies deterministic particle output for a fixed seed.
- Reduced-motion test verifies animations are disabled or simplified.
- Visual story shows fixed and contained modes.

## Accessibility Requirements

All marketing kit components must meet the same accessibility bar as app components.

- Background/orb/logo visuals are decorative by default and must not pollute screen reader output.
- `LogoMark` supports an accessible label when used as branding.
- `DisplayText` renders semantic headings using `as`.
- Gradient text must have high-contrast fallback.
- Buttons keep visible focus.
- Copy interactions announce copied state.
- Reduced-motion disables drifting/pulsing/particle animations.
- No content depends only on animation.
- No rapid flashing.

## SSR And Hydration Requirements

The homepage currently relies on client-side motion. The package components must be safe in React/Next environments.

- No direct `window`, `document`, or `navigator` access during render.
- Clipboard access only inside guarded event handlers.
- Particle positions use seeded deterministic generation.
- Random IDs use stable deterministic IDs or React `useId`.
- Motion values should initialize consistently server/client.
- Components should work in Next App Router.

## Performance Requirements

The marketing kit must look premium without introducing avoidable GPU or memory issues.

- `AuroraBackground` should use CSS gradients and transforms, not per-frame canvas by default.
- Particle count defaults must be conservative.
- `motion="full"` can be opt-in; default should be performant.
- Reduced-motion and low-power modes should reduce animation work.
- No `requestAnimationFrame` loops unless required and cleaned up.
- No `gl.readPixels()`-style per-frame stalls.
- Components should not significantly increase base bundle for users who do not import marketing kit components.

## Risks

### Risk: Marketing Kit Bloats Core Bundle

Mitigation:

- Keep components tree-shakeable.
- Avoid importing heavy dependencies.
- Prefer CSS gradients over canvas/WebGL.
- Confirm package export tests still pass.

### Risk: Homepage Loses Polish After Conversion

Mitigation:

- Use the existing homepage as the visual baseline.
- Capture before/after screenshots.
- Allow page-specific layout CSS, but not fake component chrome.

### Risk: API Surface Becomes Too Marketing-Specific

Mitigation:

- Expose primitives, not the exact homepage section.
- Keep copy/content/layout in website.
- Package reusable atoms/compositions only.

### Risk: Accessibility Regression From Decorative Visuals

Mitigation:

- Decorative visuals are `aria-hidden`.
- Text components are semantic.
- Reduced-motion required.
- Contrast snapshots reviewed.
